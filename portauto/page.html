1. 테스트 자동화가 필요한 이유

  • 속도: 배포후 리그레이션 테스트 시간 감소. 반복테스트에 대한 시간 경감
  • 안정성: 수동테스트에대한 오류/누락 최소화, 일관된 검증
  • 비용 절감: 초기 자동화 투자 → 배포시 테스트 검증 비용 급감
  • 개발자 경험: 즉각 피드백로 리팩토링/확장 가능
  • 확장성: 필요한 케이스에 의한 테스트 추가 확장용이

2. 테스트 도구(프레임워크·언어·아키텍처)

언어 · 프레임워크

  • 언어: Python 3.11+
  • 프레임워크: Selenium, Appium and Playwright, Appium
  • 테스트플랫폼: Web, AOS, iOS
  • E2E(웹/모바일): 사용자의 실제 행동 흐름을 처음부터 끝까지 시뮬레이션하여 전체 시스템이 올바르게 동작하는지 확인

아키텍처 원칙

  • 테스트 피라미드: Unit > Integration > E2E
  • Page Object/Screenplay 패턴
  • 폴더 구조 예시:

    devices/

3. Python 선정 이유와 장단점

장점

  • 가독성/생산성 높음, 풍부한 생태계(pytest/Playwright)
  • 범용성(백엔드·데이터·QA 협업), 크로스플랫폼

단점 및 보완

  • 실행 속도, GIL 제약 → pytest -n 병렬/분산, E2E 최소화
  • 엄격 타입 부재 → mypy 타입, ruff 린트
  • I/O 바운드 최적화 → 비동기/캐시 전략

4. 테스트케이스 구현 시 허들

  • 플래키: 비결정적 대기/애니메이션/시간 → 명시적 대기, 안정 셀렉터, 가상시계, 제한적 리트라이
  • 테스트 데이터: 중복/오염/순서 의존 → 데이터 팩토리, 케이스별 생성·정리, 샌드박스
  • 외부 의존성: 타사 API/레이트리밋 → 계약 테스트, 스텁/모킹, testcontainers
  • 인증/보안: 토큰 만료/2FA → 시드 계정, 토큰 발급 API, 시크릿 관리
  • 비동기/실시간: 큐/웹소켓 동기화 → 폴링·훅, 컨슈머 테스트
  • 가시성: 실패 원인 파악 → Allure 단계/스크린샷/네트워크 로그

5. 테스트 시스템: Jenkins 활용

Checkout → Lint/Type → Unit → Integration(Compose) → E2E(병렬) → 리포트/아티팩트 → 알림

pipeline {
  agent any
  stages {
    stage('Setup') {
      steps { sh 'python -m venv .venv && . .venv/bin/activate && pip install -U pip -r requirements.txt' }
    }
    stage('Lint & Type') {
      steps { sh '. .venv/bin/activate && ruff check . && mypy .' }
    }
    stage('Unit & Integration') {
      steps {
        sh '. .venv/bin/activate && pytest -q tests/unit --cov=src --cov-report=xml'
        sh 'docker compose -f docker-compose.test.yml up -d'
        sh '. .venv/bin/activate && pytest -q tests/integration'
      }
      post { always { sh 'docker compose -f docker-compose.test.yml down -v' } }
    }
    stage('E2E (Parallel)') {
      parallel {
        stage('Web') { steps { sh '. .venv/bin/activate && pytest -q tests/e2e -m web --alluredir=allure' } }
        stage('Mobile') { steps { sh '. .venv/bin/activate && pytest -q tests/e2e -m mobile --alluredir=allure' } }
      }
    }
  }
  post {
    always { archiveArtifacts artifacts: 'allure/**,coverage.xml', fingerprint: true }
  }
}

6. 대표 테스트케이스 진행 순서 (커머스 결제 E2E)

  1. 로그인 → 세션/쿠키 검증
  2. 상품 검색 → 정렬/필터 동작 확인
  3. 장바구니 담기 → 수량/합계 금액 검증
  4. 결제 진행 → 주소/결제수단/3DS
  5. 주문 확인 → 주문번호 노출, 백엔드 상태 확인

검증 포인트: UI/네트워크 페이로드, DB 레코드, 이벤트 발행, 이메일/푸시
